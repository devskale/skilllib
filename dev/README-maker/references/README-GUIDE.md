### Key Sections by Project Type

**All Projects:**
- Title with one-line description
- Badges (CI, coverage, license, version)
- Quick start example
- Installation instructions

**CLI Tools:**
- Installation (npm, brew, npx)
- Usage examples with commands
- Available flags and options
- Command reference tables

**Libraries:**
- API documentation
- Configuration options
- Code examples with language annotations
- Requirements and dependencies

**Web Apps:**
- Screenshots or demo links
- Deployment instructions
- Environment variables
- Backend requirements

## Style Guidelines

**Writing:**
- Be concise - every word earns its place
- Use active voice: "Run this" not "This should be run"
- Show, don't tell: Examples > descriptions
- Front-load value - most important info first

**Visual Elements:**
- Emojis for visual hierarchy
- Code blocks with language tags
- Tables for structured data
- Badges for version/license/platform

**Progressive Disclosure:**

- Multiple paths for different users
- Copy-paste ready examples
- Error prevention with common pitfalls

## Content Checklist

**Must-Have:**
- [ ] Clear one-line description
- [ ] Installation instructions (multiple methods)
- [ ] Quick start example
- [ ] Requirements/dependencies
- [ ] Configuration options
- [ ] Troubleshooting section

**Quality:**
- [ ] Code examples tested and working
- [ ] Valid links
- [ ] Consistent formatting
- [ ] No broken markdown

## Pitfalls to Avoid

**Content:**
- Wall of text without breaks
- Vague descriptions ("powerful", "amazing")
- Missing installation steps
- Outdated examples

**Technical:**
- Broken links or images
- Untested code examples
- Missing platform requirements
- No version information

**Style:**
- Inconsistent formatting
- Poor visual hierarchy
- No emojis or visual elements
